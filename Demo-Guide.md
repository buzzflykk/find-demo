# Demo 演示手册：时光印记

> 作品集面试演示用 | 预计演示时长：8-10 分钟

---

## 演示前准备

### 环境要求

- Node.js >= 18
- 浏览器 Chrome / Edge 最新版
- 屏幕分辨率建议 >= 1440px（桌面端显示手机壳模拟器需要宽度）
- 如用窄屏，手机壳模拟器会自动隐藏改为全屏展示

### 启动命令

```bash
cd xuemai-xunzong
npm install         # 安装服务端依赖
cd client && npm install  # 安装客户端依赖
cd .. && npm run dev      # 同时启动前端 :5173 + 后端 :3001
```

访问 http://localhost:5173

### 演示账号

登录页手机号输入任意 11 位手机号，验证码自动填入 `123456`，点击登录即可。不需要真实手机号。

## 演示流程

### Part 1：登录与首页（1.5 分钟）

**操作**：
1. 打开 http://localhost:5173 → 看到手机壳模拟器中的登录页
2. 输入手机号 `13800138000` → 点击"发送验证码" → 验证码自动填入 `123456`
3. 点击"登录" → 进入首页信息流

**解说要点**：
- 手机壳模拟器是用 CSS 实现的 iPhone 框，所有页面和交互都在框内展示
- 真实时间显示、信号/WiFi/电池图标，视觉上接近真机
- 登录用了 JWT，手机号验证码流程是真实流程（演示版自动填充验证码）
- 首页瀑布流是双列卡片布局，支持无限滚动（下滑自动加载更多）

**展示重点**：
- 手机壳外观、状态栏
- 瀑布流布局
- 加载更多（下滑到页面底部触发）

---

### Part 2：发布流程 — 照片路径（2 分钟）

**操作**：
1. 点击右下角 "+" 按钮 → 进入发布流程 Step 1
2. Step 1 默认展示上传区域 → 点击"从相册选择"（演示版 Mock，直接返回模拟照片）
3. 上传完成后 → 点击底部"开始 AI 处理"→ 进入 Step 2 AI 处理动画
4. Step 2 有进度条 + 分步提示 → 自动进入 Step 3 结果展示
5. Step 3 展示 AI 提取的关键信息 → 点击"下一步"
6. Step 4 补充描述表单 → 填写姓名"王小明"、失联时间"2005年"、地点"北京" → 点击"下一步"
7. Step 5 海报编辑 → 展示海报预览 → 可切换模板 → 点击"下一步"
8. Step 6 发布确认 → 显示发布选项 → 点击"确认发布"
9. 回到首页，新发布的寻人信息出现在列表顶部

**解说要点**：
- 6 步发布流程，顶部进度指示器显示当前位置
- AI 处理步骤有显影动画（从模糊到清晰），让等待有感知
- 海报提供了多套模板风格

**展示重点**：
- Step 1 上传交互
- Step 2 显影动画过渡
- Step 3 AI 提取结果展示
- Step 5 海报模板切换

### Part 3：发布流程 — 纯文字路径（1 分钟）

**操作**：
1. 再次进入发布流程（首页 "+" 按钮）
2. Step 1 顶部切换到"仅文字描述" Tab
3. 在文本框中输入"寻找小时候的邻居，1998 年住在福州台江区 XX 巷，女孩小名叫阿花"
4. 点击底部"下一步：补充信息" → 直接跳过 AI 处理步骤（Step 2、3）
5. 进入 Step 4 补充描述 → 快速填写 → 继续到发布

**解说要点**：
- 纯文字模式是专门为"只有模糊记忆没有照片"的场景设计的
- 系统自动跳过 AI 处理步骤，减少等待
- 没有照片/信件时也能完成完整的发布流程

**展示重点**：
- Tab 切换（媒体 / 仅文字）
- 跳步逻辑

## 典型面试问答

### Q：这个项目是你一个人做的吗？用了多久？

从产品定义到可运行 Demo，独立完成全部工作：产品需求文档、用户研究、竞品分析、交互设计、视觉设计、前后端全栈开发。时间视具体项目阶段而定。

### Q：最难的部分是什么？

发布流程的状态管理。6 步表单 + 条件跳转（纯文字模式跳过 AI 步骤）+ 每个步骤有自己的表单数据。用自定义 `usePublish` Hook 管理所有步骤的共享状态，保证数据在步骤切换间不丢失。

### Q：AI 匹配是怎么实现的？

目前是模拟数据 + 纯字符串比对。用户在搜索时，系统将输入关键词与所有寻人信息的 `target_name`、`lost_location`、`lost_type` 等字段做模糊匹配，按匹配度排序返回结果。架构上预留了模型接口，真实人脸匹配（DeepFace + ArcFace）和历史文本对比（NLP 相似度）可以作为后端服务替换。

### Q：怎么保证图片上传的性能？

前端做了图片压缩（canvas 缩放），限制上传尺寸，减少传输时间。后端用 multer + 内存存储，处理完就释放。生产环境替换为 OSS 对象存储 + 分片上传。

### Q：夜间模式是怎么做的？

CSS 自定义属性体系。定义 `--color-bg`、`--color-text` 等变量，`:root` 下赋值浅色，`.dark` 下赋值深色。`ThemeProvider` 组件管理主题状态，持久化到 `localStorage`。切换时只需切换 `<html>` 的 `.dark` class，全局同步变色。

## 演示注意事项

### Demo 限制（提前说，避免面试官以为 Bug）

- **AI 功能是模拟的** — 上传照片后不会真的调用模型处理，返回的是预设数据。但流程和交互和真实一致
- **图片 / 信件上传是模拟的** — 点击"从相册选择"会触发浏览器文件选择器，选择了照片后系统会展示它，但 AI 处理结果是模拟的
- **数据是内存存储** — SQLite 文件存储，刷新页面数据不会丢，但服务器重启后数据会重置（除非部署了持久化存储）
- **不涉及真实短信** — 验证码是自动填入的演示模式
- **所有匹配数据是预设的** — 没有真实 AI 引擎在后台运行

### 加分展示点

- 页面右下角"+""按钮（FAB）在手机壳内定位正确
- 底部导航栏贴合手机框底部
- 夜间模式切换前后视觉效果
- 瀑布流加载更多（一直下滑，出现"已加载全部"提示）
- 手机壳在小窗口下会自动平铺（<768px 隐藏手机壳）

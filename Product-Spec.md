# Product Spec：时光印记

> 版本：v1.0 | 状态：初稿 | 最后更新：2026-05-09

---

## 产品概述

「时光印记」是一款面向年轻人的旧时光寻人工具型 App，核心逻辑是：用户翻出老照片或旧信件 → AI 自动修复和提取线索 → 用户补充描述 → 生成寻人海报 → 一键发布到社交媒体 → 同时 App 内沉淀数据支持 AI 自动匹配同线索用户。

**目标用户**：
- **核心用户**：90后/00后年轻人，想找回儿时邻居、幼儿园/小学同学、已失联朋友，但只有模糊记忆或旧照片信件
- **辐射用户**：中老年群体（通过子女或自己操作），寻找老战友、老同学、知青故友

**核心价值**：把模糊的记忆变成可传播的线索，让"想找但不知道怎么找"的人有一条完整的行动路径——AI 帮你整理线索、生成海报、传播出去，然后等结果回来。

### 面试展示定位（2026-05-26 迭代）

本项目当前前端定位调整为 **C 端 / AI 产品经理面试展示型 Demo**。打开项目后默认进入手机 App Demo，不做单独作品集落地页，避免面试官先看到一屏自我介绍而不是产品本身。

Demo 的展示重点从"功能可用"升级为"产品逻辑可讲清楚"：
- 在 App 首页内直接呈现产品定位、目标用户痛点、AI 介入价值和核心行动入口。
- 2026-05-30 迭代：将产品定位、目标用户痛点、AI 介入价值和核心行动入口从 App 首页抽离为独立开屏页。开屏页参考 Apple 线下展示机的引导方式，用户点击屏幕任意位置即可进入可交互 App。
- 用轻量信息模块展示 C 端转化路径：上传线索 → AI 增强 → 生成可传播海报 → 社交扩散 → 站内匹配。
- 保留完整 App 原型路径，面试时可以直接点击发布流程、探索页、消息页和个人页进行演示。
- AI 能力仍使用 Mock 方式展示，不接入真实模型；前端需要明确让面试官感知这是原型演示版，而不是生产环境冒充真实能力。
- 不以个人求职海报作为首屏重点，个人能力通过产品结构、AI 流程、用户洞察、指标和合规意识间接呈现。

## 应用场景

- **翻出老照片找人**：小林翻到一张 2005 年小学毕业照，指着第二排第三个说"我记得他叫李明，但我忘了他是哪个中学了"。拍照上传，AI 修图提亮人脸，他补充"XX 小学 2005 届毕业后失联"，AI 生成寻人海报发布到朋友圈。老家有人看到后转发，李明的前同事看见，牵上线。

- **旧信件里的模糊线索**：阿婷收拾奶奶遗物翻出一封 1998 年的信，落款是"广州陈阿姨"。拍照上传，AI 识别手写文字提取出"建设二路""陈秀兰"等关键词，她补充"奶奶叫王秀珍，大概 2000 年后失联"，海报发到小红书后被陈阿姨的女儿看到。

- **儿时玩伴只有小名**：大军记得小时候住福州台江区 XX 巷，邻居家女孩小名叫"阿花"，1998 年搬走后失联。他输入"福州台江区 XX 巷，约 1995-1998，邻居女孩小名阿花"，AI 匹配到另一位用户发布过同巷的寻人信息，系统提醒二人联系。

- **找到后人走茶凉**：小美通过 App 找到了失联 20 年的小学闺蜜，二人在 App 里互发私信相认。小美在 App 里标记"已找到"，这条寻人信息自动从公共匹配池移除，存入她的个人回忆档案。

- **基础的族谱记录**（第二期）：阿杰回老家翻了族谱，把曾祖父到自己的五代人信息输入 App，自动生成一个族谱树。但他发现中间有一代人对不上，先放着，等以后回老家查证再补。

## 功能需求

### 核心功能

- **老照片上传与 AI 修复**
  用户从相册选择或拍摄老照片 → 系统上传至云端 → AI 调用 Real-ESRGAN + GFPGAN 模型进行画质修复和人脸增强 → 返回高清修复后的照片给用户预览
  
- **信件上传与文字提取**
  用户拍摄或上传旧信件照片 → 系统上传至云端 → AI 调用 OCR 模型（PP-OCRv5 / GLM-OCR）识别手写/印刷文字 → 返回识别文本 → AI 自动提取关键信息（人名、地名、时间、联系方式等）→ 标注展示给用户确认/修改

- **寻人线索描述**
  用户补充文字描述：目标人物姓名（或昵称）、失联时间地点、特征描述、其他线索 → 系统与 AI 提取的关键信息合并为结构化线索数据

- **AI 生成寻人海报**
  用户点击生成海报 → AI 将修复后的照片 + 提取的关键信息 + 用户描述组合为一张寻人海报 → 提供多套模板供用户选择 → 用户确认后生成最终版

- **一键发布到社交媒体**
  用户点击分享 → 系统生成社交平台适配的分享卡片 → 呼起系统分享面板支持微信/朋友圈/微博/小红书等 → 分享后返回 App

- **内容沉淀与 AI 匹配**
  每条寻人信息自动进入 App 公共池 → AI 引擎对所有活跃寻人信息进行交叉分析（比对时间、地点、人物名称相似度、照片人脸相似度）→ 发现高匹配度线索时向双方推送通知 → 匹配双方可通过站内私信沟通

- **已找到标记与下架**
  用户标记寻人信息为"已找到" → 信息从公共匹配池移除不再曝光 → 信息保留在该用户的个人存档中，可随时查看

- **站内私信**
  匹配双方可通过站内私信进行沟通 → 支持文字、图片 → 私信仅对匹配成功的双方开放

### 辅助功能

- **个人寻人档案**
  用户主页展示所有历史寻人记录 → 每条记录显示：发布状态（寻人中/已找到）、发布时间、浏览量/匹配数 → 已找到的记录独立归档

- **通知中心**
  系统通知：AI 匹配结果、私信提醒、海报被查看/转发等 → 通知推送

- **基础族谱树（第二期入口）**
  Tab 栏占位入口 → 当前显示"即将上线"页面 → 第二期实现手动输入家人信息自动生成族谱树

- **账号管理**
  手机号验证码登录 → 个人资料设置（昵称、头像）→ 退出登录/注销账号

- **会员中心**
  展示当前订阅状态 → 提供订阅方案选择（连续包月/包年）→ 展示免费剩余次数 → 历史订单

## AI 能力需求

| 能力类型 | 用途说明 | 触发时机 |
|---------|---------|---------|
| 图像修复与增强 | 老照片画质修复（清晰度、去噪、色彩还原）+ 人脸细节增强 | 用户上传老照片后自动触发 |
| 文字识别（OCR） | 手写/印刷信件文字的识别与数字化，支持中英文及竖排 | 用户上传信件照片后自动触发 |
| 信息提取（NLP） | 从识别文本中自动提取人名、地名、时间、联系方式等关键寻人信息 | OCR 完成后自动触发 |
| 人脸相似度匹配 | 跨年龄人脸比对，用于匹配同一人的不同时期照片 | 用户发布寻人信息后，AI 引擎交叉比对 |
| 智能匹配算法 | 基于时间、地点、人物名称、文本相似度的多维度线索匹配 | 用户发布寻人信息后持续运行 |
| 海报生成 | 将照片、关键线索、用户描述自动合成为寻人海报 | 用户点击"生成海报"时触发 |

## 用户使用流程

### 完整路径：第一次使用

1. 下载 App → 手机号验证码登录
2. 首页点击"发布新寻人"
3. 选择"上传老照片"、"上传信件"或"仅文字描述"
4. 如上传了照片/信件 → 等待 AI 处理（照片修复 / 文字识别）；如仅文字 → 直接进入补充信息
5. 查看 AI 提取的关键信息，确认或修改（仅照片/信件路径）
6. 补充文字描述：目标人物姓名、失联时间地点、其他线索
7. 点击"生成寻人海报"，选择模板
8. 确认海报效果
9. 选择"发布到 App 公共池"+"同步分享到社交媒体"
10. 分享完成后回到 App 查看寻人记录

### 收到匹配通知

1. 收到推送通知"发现可能的线索"
2. 打开 App 查看匹配详情（匹配度、相似线索）
3. 如认为匹配 → 发起站内私信联系对方
4. 双方确认后 → 标记"已找到"
5. 寻人信息从公共池移除，存入个人档案

### 搜索浏览

1. 进入"探索"页 → 浏览公共池中的寻人信息
2. 按地区/时间段/关键词筛选
3. 如发现有线索可帮 → 通过私信联系发布者
4. 或分享到社交网络帮助扩散

## UI 布局

### 整体架构
底部 Tab 栏导航（4 个标签页），顶部状态栏区域。

### 底部 Tab 栏

| Tab | 图标/名称 | 内容 |
|-----|----------|------|
| 首页 | 时光机 | 核心入口，发布寻人 + 推荐信息流 |
| 探索 | 发现 | 公共寻人信息池，搜索/筛选/浏览 |
| 消息 | 私信 | 站内私信列表 + 通知中心 |
| 我的 | 个人 | 个人档案、会员中心、设置、族谱入口 |

### 各页面布局

**开屏页**
- 全屏展示产品定位："把模糊记忆变成可传播线索"
- 展示核心路径、AI 节点、MVP 边界和 4 步 AI 闭环
- 参考 Apple 线下展示机开屏体验，弱化按钮感，用户点击屏幕任意位置进入 App 首页

**首页（时光机）**
- 顶部：App 名称"时光印记" + 通知图标
- 信息流顶部提供「本地 / 全部」分段切换：
  - 本地：优先展示与用户当前城市/地区相关的寻人线索，用户可点击开启定位，系统自动匹配地点。
  - 全部：展示公共池内所有可见寻人线索。
  - 演示版中定位使用 Mock 城市，不调用真实定位权限。
- 中部：推荐信息流（按时间倒序展示公共池中的寻人卡片）；无真实数据或接口异常时展示内置演示案例，保证面试现场不空屏
- 卡片内容：缩略图 + 标题 + 发布时间 + 匹配数

**发布流程页**
- Step 1：媒体上传区（照片/信件，支持相册选择或拍照）+ 「仅文字描述」切换模式（输入文字描述失联信息）
- Step 2：AI 处理中动画（进度条 + 分步提示；仅上传了照片/信件的路径需要此步骤）
- Step 3：AI 提取结果展示（照片修复对比 / 文字识别结果 + 关键信息标注；仅照片/信件路径需要）
- Step 4：补充描述表单（姓名、时间、地点、特征等字段）
- Step 5：海报编辑页（模板选择 + 预览）
- Step 6：发布确认（App 内发布 + 社交媒体分享选项）

**探索页**
- 顶部：搜索栏（关键词搜索）
- 筛选栏：地区筛选、时间范围、失联类型
- 信息列表：寻人卡片瀑布流

**消息页**
- Tab 切换："私信" / "通知"
- 私信列表：匹配成功的对话
- 通知列表：匹配结果、系统消息

**个人页**
- 顶部：用户头像、昵称、会员状态
- 功能区：我的寻人、已找到档案、族谱入口（占位）、会员中心
- 底部：设置（账号管理、隐私、关于）

## 技术方向

| 维度 | 选择 | 理由 |
|------|------|------|
| 产品类型 | 移动 App | 拍照/相册上传是核心操作，天然适合移动端 |
| 推荐技术栈 | Flutter（跨平台前端）+ Go/Node.js（后端）+ PostgreSQL + Redis + 阿里云OSS | Flutter 一套代码覆盖 iOS/Android，开发效率高；后端需处理图片/文件上传、AI 模型调度、消息推送 |
| AI 模型 | Real-ESRGAN + GFPGAN（修复）、PP-OCRv5 / GLM-OCR（OCR）、CompreFace/DeepFace+ArcFace（人脸匹配） | 成熟开源方案，可部署在云端 GPU 服务，移动端通过 API 调用 |
| 数据存储 | 云端（阿里云 OSS + PostgreSQL） | 照片/信件存储在 OSS，结构化数据存储在 PostgreSQL |
| 账号体系 | 手机号验证码（阿里云/腾讯云 SMS） | 降低注册门槛，合规要求 |
| 消息推送 | 极光推送 / 阿里云移动推送 | 匹配通知、私信提醒 |
| 部署方式 | App Store + 各大安卓应用商店 | 面向 C 端用户必须全渠道覆盖 |

## 技术说明

### 外部依赖
- **AI 图像修复**：Real-ESRGAN + GFPGAN，需 GPU 服务器，单次处理预估 3-10 秒
- **OCR 识别**：PP-OCRv5 / GLM-OCR，CPU 可运行，单次识别预估 1-3 秒
- **人脸匹配**：DeepFace + ArcFace，需 GPU，每次入库/比对预估 2-5 秒
- **短信服务**：阿里云/腾讯云短信验证码，约 0.05 元/条
- **对象存储**：阿里云 OSS，图片存储按量计费
- **地图/位置服务**：高德/百度地图 API，用于地点解析和展示

### 安全与隐私
- 用户上传的原始照片和信件加密存储
- 寻人信息中的联系方式仅在匹配双方确认后通过私信交换
- 用户可随时删除自己的数据和账号
- 遵循《个人信息保护法》相关要求
